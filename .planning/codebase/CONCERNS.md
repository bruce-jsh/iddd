# Codebase Concerns

**Analysis Date:** 2026-04-06

## Tech Debt

**hasMatchingRule is a stub:**
- Issue: The `hasMatchingRule` function in `src/hooks/rule-check.ts` (lines 82-94) does not actually verify that a specific validation detection has a matching business rule. It merely checks if the file `docs/business-rules.md` contains the string `BR-` and is longer than 200 characters. Any file meeting those two conditions causes the function to return `true` for every detection, making the rule-check hook essentially a no-op when a business-rules file exists.
- Files: `src/hooks/rule-check.ts`
- Impact: The rule-check hook will silently pass all validations as long as a business-rules.md file with at least one BR-xxx entry exists, regardless of whether the specific validation pattern is actually registered.
- Fix approach: Implement proper matching logic that correlates the `ValidationDetection` (file, pattern, content) with specific BR-xxx entries. Parse the business-rules.md table and match by file path pattern or validation framework.

**Codex hooks.json references non-existent iddd-rule-check.js:**

Only `iddd-schema-drift.js` and `iddd-auto-audit.js` are built. The built hooks directory (`templates/hooks/`) confirms `iddd-rule-check.js` does not exist.
- Files: `templates/.codex/hooks.json`, `scripts/build-hooks.ts`
- Impact: Codex users who rely on the `PostToolUse` hook for business-rule checking will get a silent `node` error (file not found) on every tool use. This may also cause Codex to report hook failures depending on its error handling.
- Fix approach: Either build an `iddd-rule-check.js` entry point via esbuild (create `src/hooks/codex-rule-check-entry.ts`), or remove the non-functional hook from `templates/.codex/hooks.json`.

**Google Antigravity platform not implemented:**

- Files: `src/init.ts` (lines 174-178), `templates/`
- Impact: Antigravity platform support is advertised but absent. Users expecting Antigravity integration will find nothing.
- Fix approach: Either implement Antigravity support (detection, hook templates, skill symlinks) or remove the claim from documentation.

**No linting or formatting tooling:**

**No CI/CD pipeline:**

**dist/ is in .gitignore but templates/hooks/ contains built artifacts that ARE committed:**
- Issue: The bundled JavaScript files in `templates/hooks/` (pre-commit, post-commit, iddd-schema-drift.js, iddd-auto-audit.js) are committed to git and published to npm. These are build outputs generated by `scripts/build-hooks.ts` via esbuild. If the source changes but `npm run build` is not run before commit, the template hooks become stale.
- Files: `templates/hooks/pre-commit`, `templates/hooks/post-commit`, `templates/hooks/iddd-schema-drift.js`, `templates/hooks/iddd-auto-audit.js`
- Impact: Stale hook bundles shipped to users could have different behavior than the source code suggests. The `prepublishOnly` script runs build before publish, but commits in between could have mismatched bundles.
- Fix approach: Either add the build step to a pre-commit hook for this project, or add `templates/hooks/*.js` and `templates/hooks/pre-commit` and `templates/hooks/post-commit` to `.gitignore` and ensure they are generated at publish time only. The `"files"` field in `package.json` already includes `templates/`.

## Known Bugs

**Potential division by zero in UI gate coverage calculation:**
- Symptoms: If `entityCoverage` string is `"0/0 (0%)"` (the default when no coverage data exists), the regex captures `0` and `0`, leading to `(0 / 0) * 100` which produces `NaN` in JavaScript. `NaN >= 100` evaluates to `false`, so the pillar fails, which is arguably correct behavior but the `NaN` propagation is unintentional.
- Files: `src/preview/ui-gate.ts` (line 205)
- Trigger: Run `runSevenPillarGate` with a `UiStructureData` whose `entityCoverage` is `"0/0 (0%)"`.
- Workaround: The `NaN >= 100` comparison coincidentally produces the correct `BLOCKED` result, but it is a latent bug that could cause issues if the value is used in arithmetic elsewhere.

**Naive pluralization logic:**
- Symptoms: The `pluralize` function in `src/preview/mockup-generator.ts` (lines 53-58) handles only three cases: words ending in `s` (adds `es`), consonant+y (changes to `ies`), and everything else (adds `s`). Common English nouns like "Child" -> "childs" (should be "children"), "Person" -> "persons" (should be "people"), "Address" -> "addresses" (would produce "addresseses").
- Files: `src/preview/mockup-generator.ts` (lines 53-58)
- Trigger: Use entities with irregular plural forms.
- Workaround: Users must manually edit generated screen URLs.

**escapeHtml does not escape single quotes:**
- Symptoms: The `escapeHtml` function in `src/preview/renderer.ts` (lines 165-171) escapes `&`, `<`, `>`, and `"` but not single quotes (`'`). Since some generated HTML uses single-quoted attribute contexts (e.g., the `onclick` handler in `src/preview/mockup-generator.ts` line 473), entity names containing single quotes could break the HTML or allow XSS in the local preview.
- Files: `src/preview/renderer.ts` (lines 165-171), `src/preview/mockup-generator.ts` (line 473)
- Trigger: An entity name or attribute containing a single quote character.
- Workaround: Avoid single quotes in entity/attribute names.

## Security Considerations

**Shell injection via getCachedFileContent:**
- Risk: The `getCachedFileContent` function in `src/hooks/shared.ts` (line 56) passes `filePath` directly into a shell command via `execSync(\`git show :${filePath}\`)`. If a staged file has a specially crafted name containing shell metacharacters (e.g., `; rm -rf /`), it could execute arbitrary commands.
- Files: `src/hooks/shared.ts` (line 56)
- Current mitigation: The `filePath` comes from `git diff --cached --name-status` output which limits realistic exploitation. However, git does allow filenames with special characters.
- Recommendations: Use `execFileSync('git', ['show', `:${filePath}`])` instead of `execSync` with string interpolation to avoid shell interpretation entirely.

**Shell injection via commandExists:**
- Risk: The `commandExists` function in `src/init.ts` (line 182) uses `execSync(\`which ${cmd}\`)`. While `cmd` is currently hardcoded to `'claude'` or `'codex'`, the function signature accepts any string. If exposed to user input in the future, it becomes a shell injection vector.
- Files: `src/init.ts` (line 182)
- Current mitigation: Only called with hardcoded string literals. No user input reaches this function currently.
- Recommendations: Use `execFileSync('which', [cmd])` or the `which` npm package to eliminate the shell injection surface.

**Preview server binds to all interfaces on port 0:**
- Risk: The preview server in `src/preview/server.ts` (line 44) listens on `0` (random port) but on all network interfaces by default. Any device on the local network can access the preview.
- Files: `src/preview/server.ts` (line 44)
- Current mitigation: None. The server is short-lived (user presses Enter to close) and serves only generated HTML.
- Recommendations: Bind explicitly to `127.0.0.1` by changing `server.listen(0)` to `server.listen(0, '127.0.0.1')`.

**No input validation on hook JSON from stdin:**
- Risk: The `parseHookInput` function in `src/hooks/shared.ts` (lines 28-41) parses arbitrary JSON from stdin. While it does null-check for required fields, it performs no size limits or schema validation. A malformed or extremely large JSON payload could cause memory issues.
- Files: `src/hooks/shared.ts` (lines 28-41)
- Current mitigation: The hook is invoked by Claude Code's hook API which controls the input format.
- Recommendations: Add a maximum buffer size check when reading stdin chunks in `src/hooks/claude-pretool-entry.ts` (lines 10-14).

## Performance Bottlenecks

**Synchronous execSync calls in hook paths:**
- Problem: Multiple functions use `execSync` which blocks the Node.js event loop: `getCachedDiff` (line 45), `getCachedFileContent` (line 56), `commandExists` (line 182), and `logSkip` (line 138) in various files.
- Files: `src/hooks/shared.ts` (lines 43-51, 54-59, 136-139), `src/init.ts` (lines 180-186)
- Cause: `execSync` is synchronous and blocks the process. For git hooks (pre-commit, post-commit) this is acceptable since they run in sequence. For the init flow, it adds latency to platform detection.
- Improvement path: Replace with async `execFile` from `child_process/promises` where possible, especially in `src/init.ts`. For hooks that need synchronous behavior, `execFileSync` (without shell) is still preferable for security.

**readFileSync in banner function:**
- Problem: The `banner` function in `src/utils/ascii.ts` (line 53) uses synchronous `readFileSync` to load the banner asset file. This blocks the event loop during CLI startup.
- Files: `src/utils/ascii.ts` (line 53)
- Cause: The function is synchronous by design to simplify the CLI banner display.
- Improvement path: Convert to async and `await` it in the CLI entry point, or accept the minor blocking cost since it runs once at startup.

## Fragile Areas

**Relative path resolution via import.meta.dirname:**
- Files: `src/init.ts` (line 36), `src/utils/ascii.ts` (line 52)
- Why fragile: Both files use `import.meta.dirname` with relative `..` traversal to locate the `templates/` and `assets/` directories. The comment in `init.ts` says "When compiled, runs from dist/src/, so go up 2 levels to project root." If the directory structure changes (e.g., moving compiled output or restructuring), these paths silently break.
- Safe modification: If changing directory layout, update all `import.meta.dirname` path calculations. Search for `import.meta.dirname` across the codebase.
- Test coverage: `tests/init.test.ts` bypasses this by injecting `templatesDir`. The actual resolution path is not tested.

**Markdown table parsing in ui-gate.ts:**
- Files: `src/preview/ui-gate.ts` (lines 78-103)
- Why fragile: The `parseMarkdownTable` function relies on exact header patterns (e.g., `/^\|\s*Screen\s*\|/`) and assumes tables have pipe-delimited columns with a separator row containing dashes. Any deviation in formatting (extra whitespace, different header text, missing separator row) causes silent parse failures returning empty arrays.
- Safe modification: Add defensive logging when tables are not found. Consider using a proper markdown parser if complexity grows.
- Test coverage: Tests in `tests/preview/ui-gate.test.ts` cover the happy path but do not test malformed or slightly-off table formatting.

**Hook injection appends without deduplication:**
- Files: `src/init.ts` (lines 150-165)
- Why fragile: The `injectClaudeHooks` function appends hooks to `settings.local.json` using the spread operator (`...(hooks['PreToolUse'] ?? [])`) but never checks for duplicates. Running `initProject` with `overwrite: true` multiple times appends duplicate hook entries, resulting in the same hook running multiple times per event.
- Safe modification: Before appending, filter existing entries that match the IDDD hook command pattern. Deduplicate by `command` field.
- Test coverage: No test covers the duplicate injection scenario. `tests/init.test.ts` does not verify the contents of `settings.local.json`.

**matchGlob is a simplified glob implementation:**
- Files: `src/hooks/shared.ts` (lines 73-83)
- Why fragile: The custom glob matcher converts patterns to regex with simple string replacements. It handles `**` and `*` but does not support `?`, `[...]` character classes, or brace expansion `{a,b}`. Edge cases with nested `**` patterns or patterns containing regex-special characters (e.g., `+`, `(`, `)`) can produce incorrect regex.
- Safe modification: If more complex patterns are needed, use a proper glob library like `minimatch` or `picomatch`. For now, only extend the simple patterns.
- Test coverage: `tests/hooks/shared.test.ts` covers basic patterns but no edge cases with regex-special characters in file paths.

## Scaling Limits

**Single-file commit counter:**
- Current capacity: The commit counter in `.iddd/commit-count` is a plain text file storing a single integer. Concurrent git hooks or parallel commits could race on read/write.
- Limit: File-based locking is absent; two simultaneous commits could read the same count and both increment to the same value, losing a count.
- Scaling path: For the current use case (triggering audit every N commits), a lost count is low-impact. If precision matters, use file locking (`flock`) or an atomic increment approach.

## Dependencies at Risk

**Zero runtime dependencies:**
- Risk: None currently. The package has zero production `dependencies`, relying entirely on Node.js built-in modules. This is a significant strength -- no supply chain risk from transitive dependencies.
- Impact: N/A

## Missing Critical Features

**No engines field in package.json:**
- Problem: The codebase uses `import.meta.dirname` which requires Node.js >= 21.2.0 (or >= 20.11.0 with `--experimental-vm-modules`). The target in `tsconfig.json` is `ES2022` and esbuild targets `node18`. There is no `engines` field in `package.json` to enforce a minimum Node.js version.
- Blocks: Users on Node.js 18 or 20 (LTS) may encounter cryptic runtime errors when `import.meta.dirname` is undefined.

**No --help or --version CLI flags:**
- Problem: The CLI entry point `bin/cli.ts` has no help text, version flag, or usage documentation. Running `id3-cli --help` or `id3-cli --version` does nothing useful -- it just runs init on the current directory.
- Blocks: Users cannot discover CLI usage without reading documentation.

**No Windows path separator handling:**
- Problem: While `noSymlink` mode exists as a "Windows fallback" for file copying, the `matchGlob` function in `src/hooks/shared.ts` uses `/` as the path separator in its regex. Windows file paths use `\` which will not match.
- Blocks: Git hooks (schema-drift, rule-check) will silently fail to detect matching files on Windows, making the hooks non-functional.
