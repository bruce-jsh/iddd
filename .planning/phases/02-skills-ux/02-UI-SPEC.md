# Phase 2 -- UI Design Contract

> Visual and interaction contract for terminal-rendered ASCII output in `/id3-start` and `/id3-clear` SKILL.md files. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Medium:** Terminal text output rendered by Claude Code. No web framework, no CSS, no browser rendering. All "UI" is ASCII art and plain text printed to the terminal.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (terminal ASCII art) |
| Icon library | Unicode box-drawing + status symbols |
| Font | Terminal monospace (user's default) |

---

## Spacing Scale

Terminal spacing uses character counts, not pixels. All values are column widths.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 1 char | Minimum gap between inline elements |
| sm | 2 chars | Gap between status symbol and label |
| md | 4 chars | Indentation for content inside boxes |
| lg | 6 chars | Indentation for nested sub-sections |
| gutter | 4 chars (`───>`) | Arrow connector between Phase boxes |
| box-inner-pad | 1 line top + 1 line bottom | Vertical padding inside each box |
| section-break | 1 blank line | Between dashboard sections (header, pipeline, progress bar, status) |

Exceptions: none

---

## Typography

Terminal output has no font-size control. "Typography" maps to text formatting conventions.

| Role | Format | Usage |
|------|--------|-------|
| Banner title | Block ASCII art (existing `assets/banner.txt` pattern) | Welcome message on first-time auto-setup only |
| Section header | ALL CAPS, no decoration | Dashboard section labels (not currently used -- dashboard uses box titles) |
| Phase label | Title Case, weight normal | Phase names inside pipeline boxes: "Entities", "Info Model", "UI Design", "Build" |
| Status value | Monospace numeric/symbol | Version numbers, counts inside boxes: "v1.0", "12" |
| Body text | Sentence case | Guidance messages, routing announcements, clear warnings |
| Emphasis | Backtick-wrapped (`text`) for file paths and commands; bold not available in terminal | File references, command references |

---

## Color

Terminal color is not controlled by SKILL.md output (Claude renders plain text). The color contract defines semantic meaning through symbols instead.

| Role | Symbol | Meaning |
|------|--------|---------|
| Complete | `checkmark` (literal checkmark character) | Phase is complete |
| In progress | `diamond` (literal diamond character) | Phase currently actionable |
| Not started | `circle` (literal circle character) | Phase not yet reachable |
| Progress filled | `full-block` (literal full-block character) | Filled portion of progress bar |
| Progress empty | `light-shade` (literal light-shade character) | Empty portion of progress bar |
| Warning | `warning-triangle` (literal warning character) | Destructive action warning in `/id3-clear` |
| Connector | `long-dash` + `>` arrow | Flow direction between Phase boxes |

Accent reserved for: The diamond symbol marks ONLY the single currently-actionable phase. Never use diamond for multiple phases simultaneously.

## ASCII Art Component Library

### Component 1: Welcome Banner

Used on: First-time auto-setup (ROUT-01), before `npx id3-cli .` runs.

```
Source: assets/banner.txt (existing pattern with double-line box: top-left-double through bottom-right-double)
```

Reuse the existing `banner.txt` style. The SKILL.md must instruct Claude to render the banner using double-line box characters matching the established pattern. Include the tagline "Your information model is your harness." inside the banner.

### Component 2: Dashboard Header

Used on: Every `/id3-start` invocation (UX-01).

```
top-left-double + repeated-double-horizontal + top-right-double
vertical-double  IDDD - Information Design-Driven Development           vertical-double
vertical-double  Your information model is your harness.                vertical-double
bottom-left-double + repeated-double-horizontal + bottom-right-double
```

Width: 60 characters inner content. Double-line box characters matching `banner.txt` style.

### Component 3: Phase Pipeline

Used on: Every `/id3-start` invocation (UX-01).

```
  top-left-single + repeated-single-horizontal + top-right-single    top-left-single + repeated-single-horizontal + top-right-single    top-left-single + repeated-single-horizontal + top-right-single    top-left-single + repeated-single-horizontal + top-right-single
  vertical-single  Phase 0/1   vertical-single dash-dash-dash-arrow vertical-single   Phase 2    vertical-single dash-dash-dash-arrow vertical-single  Phase 2.5   vertical-single dash-dash-dash-arrow vertical-single  Phase 3-5   vertical-single
  vertical-single  Entities    vertical-single    vertical-single  Info Model  vertical-single    vertical-single  UI Design   vertical-single    vertical-single    Build     vertical-single
  vertical-single    checkmark  12     vertical-single    vertical-single   checkmark  v1.0    vertical-single    vertical-single      diamond       vertical-single    vertical-single      circle       vertical-single
  bottom-left-single + repeated-single-horizontal + bottom-right-single    bottom-left-single + repeated-single-horizontal + bottom-right-single    bottom-left-single + repeated-single-horizontal + bottom-right-single    bottom-left-single + repeated-single-horizontal + bottom-right-single
```

Each box: 16 characters wide (14 inner + 2 borders). Single-line box characters. Boxes connected by `dash-dash-dash-arrow` (4-char gutter). Phase status on line 3: symbol + space + value (count, version, or nothing).

### Component 4: Progress Bar

Used on: Every `/id3-start` invocation (UX-01).

```
  repeated-single-horizontal (full width)
  full-block repeated (filled) + light-shade repeated (empty)  NN%
  repeated-single-horizontal (full width)
```

Total width: 66 characters for the bar + 4 characters for percentage label. Use single-line horizontal rule above and below.

**Progress calculation (Claude's discretion per CONTEXT.md):**
- Entities (Phase 0/1): 25% weight
- Info Model (Phase 2): 25% weight
- UI Design (Phase 2.5): 25% weight
- Build (Phase 3-5): 25% weight
- Each completed phase adds its weight. In-progress phase adds 0%.

### Component 5: Status Message

Used on: Every `/id3-start` invocation (UX-01, UX-02).

```
  > Current status: [status description]
  > Request: "[user's original request]"
```

Prefixed with `> ` (blockquote style). Two lines: current status summary + user request echo (if arguments provided). If no arguments, show suggested next action instead of request echo.

### Component 6: Clear Warning Block

Used on: `/id3-clear` invocation (CLR-02).

```
warning-triangle WARNING: This action cannot be undone.

The following IDDD files and directories will be deleted:

Directories:
  - specs/
  - docs/
  - steering/                  (warning: contains user-authored content)
  - hooks/
  - skills/
  - .claude/skills/
  - .claude/hooks/
  - .codex/skills/
  - .agents/skills/
  - .iddd/

Files:
  - CLAUDE.md
  - AGENTS.md

warning-triangle Files with user-authored content:
  - steering/product.md        (your product vision)
  - steering/data-conventions.md (your data conventions)

Proceed? [y/N]
```

Always default to N (capital N = default deny). Separate user-authored files into their own warning section with explanatory parenthetical.

## Copywriting Contract

All guidance messages adapt to the user's language (Korean for Korean speakers, English for English speakers, per CONTEXT.md). The contract below defines the English canonical form. Claude translates naturally.

| Element | Copy |
|---------|------|
| Primary CTA (no args) | "Suggested next action: [phase name]. Use `/id3-start [your request]` to begin." |
| Welcome (first-time) | "Welcome to IDDD -- Information Design-Driven Development." (followed by banner) |
| Auto-setup notice | "IDDD is not set up in this project. Setting up now..." |
| Auto-setup complete | "IDDD initialized. Here is your project dashboard:" |
| Routing announcement | "Routing to [skill name] -- [phase one-liner description]." |
| Routing with artifacts | "This phase produces: [artifact list]." |
| Empty state (no IDDD, no args) | "This project does not have IDDD set up yet. Setting up now..." |
| Empty state (IDDD exists, no args) | Show dashboard + "Suggested next action: [phase]. Use `/id3-start [request]` to proceed." |
| Ambiguous request | "Should this be a UI-only change, or does it need new data/entities? Please clarify." |
| Prerequisite not met | "Phase [N] requires [prerequisite]. Please complete [earlier phase] first." |
| Clear confirmation | "Proceed? [y/N]" |
| Clear complete | "IDDD files removed. [N] directories and [M] files deleted. Project restored to pre-IDDD state." |
| Clear cancelled | "Clear cancelled. No files were removed." |
| Clear not needed | "No IDDD files found in this project. Nothing to clear." |
| Non-project directory warning | "This directory does not appear to be a project root (no package.json, .git, etc.). Are you sure you want to set up IDDD here? [y/N]" |

## Interaction Flows

### Flow 1: `/id3-start [request]` -- First-Time Setup + Route

1. Check for IDDD markers (CLAUDE.md, specs/ directory)
2. Markers absent: check for project root indicators (package.json, .git, etc.)
3. If not a project root: show non-project directory warning, await y/N
4. Show welcome banner (Component 1)
5. Show auto-setup notice
6. Run `npx id3-cli .` (no user confirmation per CONTEXT.md decision)
7. Show auto-setup complete message
8. Read specs YAML frontmatter for project state
9. Show dashboard (Components 2+3+4+5)
10. Analyze `$ARGUMENTS` intent, route to target skill
11. Show routing announcement with phase description and artifacts (UX-03)
12. Invoke target skill

### Flow 2: `/id3-start [request]` -- Existing Project + Route

1. Check for IDDD markers -- present
2. Read specs YAML frontmatter for project state
3. Show dashboard (Components 2+3+4+5)
4. Analyze `$ARGUMENTS` intent, route to target skill
5. Show routing announcement
6. Invoke target skill

### Flow 3: `/id3-start` -- No Arguments (Status Dashboard)

1. Check for IDDD markers
2. If absent: follow Flow 1 steps 2-9, then show suggested next action instead of routing
3. If present: read specs YAML frontmatter
4. Show dashboard (Components 2+3+4+5)
5. Show suggested next action based on current state

### Flow 4: `/id3-clear` -- Project Reset

1. Check for IDDD markers (CLAUDE.md, specs/ directory)
2. If absent: show "Clear not needed" message, exit
3. Scan for all known IDDD files/directories (existence check)
4. Show clear warning block (Component 6) listing only files that actually exist
5. Await y/N confirmation (default N)
6. If N or no response: show "Clear cancelled", exit
7. If y: delete all listed files/directories
8. Show clear complete summary with counts

### Flow 5: Ambiguous Request Clarification

1. During intent analysis in Flow 1 or Flow 2
2. Request matches both UI-change and data-change patterns
3. Show ambiguous request question
4. User clarifies
5. Route based on clarification

## Routing Taxonomy (Phase Guide Reference)

| Route Target | Phase | Signal Keywords | Condition |
|-------------|-------|-----------------|-----------|
| /id3-identify-entities | 0/1 | new project, domain analysis, entity identification, what entities | No entity catalog OR user requests entity work |
| /id3-design-information | 2 | data model, information design, relationships, attributes | Entity catalog exists, user requests model work |
| /id3-design-ui | 2.5 | UI design, screen design, frontend, layout, wireframe | Data model exists, user requests UI work. Fast-path: explicit UI keywords skip entity check (ROUT-05) |
| /id3-spawn-team | 3-5 | build, implement, code, develop | UI structure exists, user requests implementation |
| /id3-info-audit | Audit | audit, verify, check model, drift | User requests audit at any point |
| /id3-preview | Preview | preview, visualize, show model | User requests preview at any point |
